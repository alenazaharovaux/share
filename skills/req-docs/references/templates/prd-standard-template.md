## 1. Problem Statement
What problem exists? Who suffers from it? What evidence do we have?

*Write concrete data, not assumptions. Link to research, support tickets, or user interviews that prove the problem exists.*

## 2. Target Users
Who will use this product? Define with specifics:
- **Primary persona:** [role, context, needs]
- **Secondary persona:** [role, context, needs]
- **Who is NOT a target user:** [explicitly exclude]

## 3. Proposed Solution
What are we building? Describe at the feature level, not the implementation level.

*Focus on what the user can do, not how the system works internally.*

## 4. Scope
| In Scope | Out of Scope |
|----------|-------------|
|          |             |

*The OUT column prevents scope creep. When in doubt, add it to OUT.*

## 5. Success Metrics
| Metric | Target | How Measured | Timeline |

|        |        |             |          |

*Without metrics, this is a wish list, not a PRD.*

## 6. Assumptions & Constraints
- **Assumptions:** what we believe to be true but haven't verified
- **Constraints:** budget, deadlines, technology limitations, team size

## 8. Non-Functional Requirements
- **Performance:** response time, throughput
- **Security:** authentication, data protection, compliance
- **Scalability:** expected load, growth projections
- **Availability:** uptime requirements, SLA

## 9. User Stories / User Flows
As a [persona], I want [action], so that [benefit].

*Include error states and edge cases, not just the happy path.*

## 10. Wireframes / Mockups
Link to designs or attach sketches. Even rough sketches reduce ambiguity.

## 11. Technical Approach
Architecture decisions, key dependencies, integrations. Keep high-level — detailed specs belong in FRD.
